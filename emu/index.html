Looks like somebody needs an upgrade!

You try to run some of my GameBoy creations - but to your dismay, you only see the following message written on the screen, in one form or another:

You are using an inaccurate or unsupported emulator. Refer to the following site for further information on emulation accuracy: zzazzdzz.github.io/emu.

This means you're using an old or buggy emulator, which incorrectly emulates certain obscure features of the GameBoy hardware. A typical GB game would never require these features to work perfectly - that's why emulators, especially ones designed with performance and speed in mind, didn't usually bother to implement such intricate details. However, my publications, especially ones related to glitching and hacking, make use of those features, and require these to function. To get them working, download a modern emulator:

  • VBA-M if you really love good old VisualBoyAdvance and want to stick with it.
  • BGB if you need powerful debugging tools and link cable support.
  • Bizhawk if you need TAS features and Lua scripting.
  • If you're on an Android device, you're out of luck. As of today (23.04.2019), I wasn't able to find any free Android emulators with decent accuracy. Some people recommend GBC.EMU or Pizzaboy.

Which emulators are bad?

  • The most notable offender in terms of emulation accuracy is the base fork of VisualBoyAdvance. This is a big one - it has become the Windows XP of emulators. At the time of its release in 2001, it was extremely innovative, implementing features that other emulators rarely had back then. It quickly became the best choice for emulating GB, GBC and GBA games, to the point that people are still using it to this date - even though VBA was released almost 18 years ago at the time of writing this! The main branch of VBA was discontinued in 2004, with the latest version being 1.8.0 - this almost 14-year-old piece of software should be abandoned by now!
  • Emulators designed for embedded devices are likely to be inaccurate, since they were primarily designed to play commercial games, where accuracy isn't a big problem. The 3DS Virtual Console is particularly known for its problems. Some emulators for mobile devices, or homebrew emulators for consoles like Nintendo Wii or DS, are also likely to have issues - they were optimized mostly for speed, not accuracy.
  • Every emulator released before 2005 is pretty much guaranteed to have accuracy issues. At the time, a lot of things were still unknown about the GameBoy hardware. People with thorough understanding of the GB internals would rather stick to the demoscene back then.

How to disable or skip this message?

You can't. Get a better emulator! It's not like I'm blocking VBA out of spite or something - the game requires high degree of emulation accuracy to function properly. This isn't something that can be "just skipped". Skipping this check would result in game crashes, hard to detect bugs, graphical issues, and would likely send you down the rabbit hole of undefined behavior.

However, if you understand the consequences of ignoring emulation accuracy, you can disable the check by patching it out. Failing the check will send the CPU into an infinite loop (jr $) - just NOP out this loop to skip the message. This intentionally requires basic reverse-engineering knowledge - it's a simple idiot test to make sure you know what you're doing.
